\section{Introduction et Contexte}
%% ============================================================

Ce projet a pour objectif de concevoir un système de \textbf{reconnaissance de gestes
en temps réel} permettant de contrôler le jeu classique \textit{Space Invaders}
directement par des mouvements de la main devant une webcam standard.

Il s'inscrit dans le cadre d'un cours d'initiation à la vision par ordinateur en
Master 2 Mathématiques, où les compétences clés à démontrer sont : la collecte de
données visuelles, la construction et l'entraînement de réseaux de neurones, et
leur intégration dans un pipeline temps réel.

\subsection{Objectifs techniques}

\begin{enumerate}
  \item Détecter et classifier 5 gestes de la main en temps réel à au moins 25 fps.
  \item Transmettre les commandes au jeu via WebSocket avec une latence inférieure à 100\,ms.
  \item Atteindre une précision de classification suffisante pour une jouabilité fluide.
\end{enumerate}

\subsection{MediaPipe HandLandmarker}

MediaPipe HandLandmarker est un modèle de détection de la main pré-entraîné par
Google, basé sur une architecture MobileNet légère. Il opère en deux étapes :
\begin{enumerate}
  \item \textbf{BlazePalm} : détecte la région de la main dans l'image en moins de 5\,ms.
  \item \textbf{Estimation des landmarks} : prédit les coordonnées $(x, y, z)$ de
        \textbf{21 points anatomiques} (poignet, phalanges proximal/médial/distal de
        chaque doigt) avec une incertitude inférieure à 2\,mm.
\end{enumerate}

En mode \texttt{VIDEO}, un tracker léger entre les frames évite la ré-détection à
chaque image, permettant un traitement à 30+ fps sur CPU sans GPU.

\subsection{Architecture logicielle — threading Python}

Le script principal \texttt{5\_jouer\_avec\_cnn.py} utilise une architecture
bi-thread pour éviter que le WebSocket bloque la capture vidéo :

\begin{lstlisting}[caption={Communication inter-threads par file de messages (extrait).}]
import queue, threading, json, websocket

commande_queue = queue.Queue(maxsize=20)

def thread_websocket(ws_uri):
    """Thread dedie a l'envoi WebSocket - ne bloque pas la video."""
    ws = websocket.WebSocket()
    ws.connect(ws_uri)
    while True:
        msg = commande_queue.get(timeout=0.5)
        ws.send(json.dumps(msg))

# Lancement du thread WS en arriere-plan
t = threading.Thread(target=thread_websocket, args=(WS_URI,), daemon=True)
t.start()

# Thread principal : webcam + MediaPipe + CNN
while True:
    ret, frame = cap.read()
    ...
    commande_queue.put_nowait({'type': geste})  # non-bloquant
\end{lstlisting}

\subsection{Hold Timer — serveur Node.js}

Le jeu JavaScript attend des événements \texttt{keydown}/\texttt{keyup}
continus, alors que Python envoie des commandes discrètes (30 fois/seconde).
Le serveur Node.js résout ce décalage avec un \textbf{hold timer de 180\,ms} :

\begin{lstlisting}[language=Java,caption={Hold timer Node.js ? maintient la touche enfonc?e tant que la commande arrive.}]
const HOLD_TIMEOUT = 180; // ms
const timers = {};

function handleCommand(type) {
  if (timers[type]) clearTimeout(timers[type]); // remet le compteur a zero
  else sendKeyDown(KEYMAP[type]);                // premiere arrivee = keydown

  timers[type] = setTimeout(() => {
    sendKeyUp(KEYMAP[type]);                     // plus de signal = keyup
    delete timers[type];
  }, HOLD_TIMEOUT);
}
\end{lstlisting}

\subsection{Approche 2 — Landmarks normalisés (Christ Bohou)}

Le script \texttt{3\_collecter\_landmarks.py} utilise MediaPipe en mode
\texttt{VIDEO} pour extraire les 21 landmarks par frame et les sauvegarde
en CSV après normalisation.

\paragraph{Normalisation des landmarks.}
Pour chaque frame, le vecteur brut $\mathbf{p} \in \mathbb{R}^{21 \times 3}$
est transformé par :

\begin{equation}
\tilde{\mathbf{p}}_i = \frac{\mathbf{p}_i - \mathbf{p}_0}
                            {\displaystyle\max_{j} \|\mathbf{p}_j - \mathbf{p}_0\|_\infty}
\end{equation}

où $\mathbf{p}_0$ est la position du poignet (landmark 0). Le résultat
$\tilde{\mathbf{p}} \in [-1, 1]^{21 \times 3}$, aplati en un vecteur de 63 valeurs,
est invariant à la position, l'échelle et la taille de la main.

\begin{lstlisting}[caption={Normalisation des landmarks ? identique ? l'entra?nement et ? l'inf?rence.}]
def normaliser(landmarks):
    pts = np.array([[lm.x, lm.y, lm.z] for lm in landmarks],
                   dtype=np.float32)
    pts -= pts[0]              # centrage sur le poignet
    s = np.max(np.abs(pts))
    if s > 0:
        pts /= s               # mise a l'echelle dans [-1, 1]
    return pts.flatten()       # vecteur de 63 valeurs
\end{lstlisting}

Environ \textbf{2\,100 échantillons} ont été collectés à raison d'une capture
automatique toutes les 300\,ms, soit environ 7 minutes de collecte par classe.
La même normalisation est appliquée à l'identique lors de l'inférence temps réel,
garantissant la cohérence entre entraînement et déploiement.

\section{Modèles de Classification}

Quatre modèles ont été expérimentés au total : deux par Christ Bohou

(MLP et CNN sur images Kaggle).

\subsection{Modèle 1 — CNN sur images brutes (Christ Bohou)}

\subsubsection{Architecture}

Un CNN à 3 blocs convolutifs avec data augmentation en ligne a été construit.

\begin{lstlisting}[caption={Architecture CNN 3 blocs (2\_entrainer\_cnn.py).}]
modele = models.Sequential([
    # Bloc 1 - detection de contours simples
    layers.Conv2D(32, (3,3), activation='relu', padding='same',
                  input_shape=(64, 64, 3)),
    layers.BatchNormalization(),
    layers.Conv2D(32, (3,3), activation='relu', padding='same'),
    layers.MaxPooling2D(2, 2),
    layers.Dropout(0.25),

    # Bloc 2 - detection de formes (doigts, paume)
    layers.Conv2D(64, (3,3), activation='relu', padding='same'),
    layers.BatchNormalization(),
    layers.Conv2D(64, (3,3), activation='relu', padding='same'),
    layers.MaxPooling2D(2, 2),
    layers.Dropout(0.25),

    # Bloc 3 - patterns complexes (configuration de la main)
    layers.Conv2D(128, (3,3), activation='relu', padding='same'),
    layers.BatchNormalization(),
    layers.MaxPooling2D(2, 2),
    layers.Dropout(0.25),

    # Tete de classification
    layers.Flatten(),
    layers.Dense(256, activation='relu'),
    layers.BatchNormalization(),
    layers.Dropout(0.5),
    layers.Dense(5, activation='softmax'),   # 5 classes
])
\end{lstlisting}

\textbf{Total paramètres entraînables : 2\,240\,037.}

\subsubsection{Data Augmentation}

Pour compenser la faible taille du dataset (2\,521 images), une augmentation
de données en ligne est appliquée à chaque batch pendant l'entraînement :

\begin{lstlisting}[caption={Pipeline d'augmentation de donn?es.}]

    layers.RandomBrightness(0.2),        # luminosite +/- 20%

Le CNN apprend à reconnaître le fond de la pièce autant que la forme de la main.
Avec 2\,521 images collectées dans un seul environnement d'éclairage, le modèle
mémorise des artefacts contextuels (couleur du mur, ombres) plutôt que la
géométrie de la main. Il en résulte une généralisation quasi nulle.

\subsection{Modèle 4 — Réseau Dense sur Landmarks (Christ Bohou) — Solution retenue}

\subsubsection{Principe et motivation}

Au lieu de traiter les pixels, on extrait d'abord la représentation géométrique
de la main (landmarks MediaPipe), puis on entraîne un réseau dense sur ces
coordonnées normalisées. La représentation obtenue est :
\begin{itemize}
  \item \textbf{invariante au fond} : aucune information de pixel
  \item \textbf{invariante à la luminosité et à la couleur de peau}
  \item \textbf{invariante à la taille} et à la distance à la caméra (normalisation)
  \item \textbf{compacte} : 63 réels au lieu de $64 \times 64 \times 3 = 12\,288$
\end{itemize}

\subsubsection{Architecture complète}

\begin{lstlisting}[caption={Architecture du r?seau dense sur landmarks (4\_entrainer\_landmarks.py).}]
model = keras.Sequential([
    keras.Input(shape=(63,)),               # 21 points x 3 coords

    keras.layers.Dense(128, activation='relu'),
    keras.layers.BatchNormalization(),
    keras.layers.Dropout(0.3),

    keras.layers.Dense(128, activation='relu'),
    keras.layers.BatchNormalization(),
    keras.layers.Dropout(0.3),

    keras.layers.Dense(64, activation='relu'),
    keras.layers.BatchNormalization(),
    keras.layers.Dropout(0.2),

    keras.layers.Dense(5, activation='softmax'),   # 5 classes
])

model.compile(
    optimizer=keras.optimizers.Adam(learning_rate=0.001),
    loss='sparse_categorical_crossentropy',
    metrics=['accuracy'],
)
\end{lstlisting}

\textbf{Total paramètres : 33\,925} — soit 66 fois moins que le CNN images.

\subsubsection{Entraînement}

\begin{lstlisting}[caption={Boucle d'entra?nement avec callbacks.}]
callbacks = [

    keras.callbacks.EarlyStopping(

        restore_best_weights=True),
    keras.callbacks.ReduceLROnPlateau(
        monitor='val_loss', factor=0.5, patience=5),

history = model.fit(
    X_train, y_train,
    epochs=50, batch_size=32,
    validation_data=(X_val, y_val),
    callbacks=callbacks,
)
\end{lstlisting}

Le modèle converge en \textbf{13 époques} sur les 50 max autorisées,
avec un temps d'entraînement total inférieur à 30 secondes sur CPU.

\subsection{Pipeline d'inférence}

La boucle d'inférence principale traite chaque frame en 3 étapes :

\begin{lstlisting}[caption={Boucle d'inf?rence temps r?el (5\_jouer\_avec\_cnn.py).}]
SEUIL_CONFIANCE = 0.70   # seuil pour envoyer une commande

while True:
    ret, frame = cap.read()
    frame = cv2.flip(frame, 1)     # miroir webcam

    # 1. Extraction des landmarks MediaPipe
    rgb    = cv2.cvtColor(frame, cv2.COLOR_BGR2RGB)
    mp_img = mp.Image(image_format=mp.ImageFormat.SRGB, data=rgb)
    result = landmarker.detect_for_video(mp_img, timestamp_ms)

    if result.hand_landmarks:
        lm_list = result.hand_landmarks[0]

        # 2. Normalisation (identique a l'entrainement)
        vecteur = normaliser(lm_list).reshape(1, -1)

        # 3. Inference CNN
        probs     = model.predict(vecteur, verbose=0)[0]
        idx       = np.argmax(probs)
        confiance = probs[idx]
        geste     = CLASSES[idx] if confiance >= SEUIL_CONFIANCE else None

        # 4. Envoi si confiance suffisante
        if geste and geste != 'NEUTRAL':
            commande_queue.put_nowait({'type': geste})
\end{lstlisting}

\section{Résultats, Limites et Analyse Critique}
%% ============================================================

\subsection{Performance en conditions réelles}

Le réseau dense sur landmarks est \textbf{robuste en conditions réelles} :
fond de pièce, éclairage variable, main de taille différente.
Les gestes LEFT, RIGHT, ENTER et NEUTRAL sont reconnus sans erreur notable.
La latence totale mesurée (capture → jeu) est de 40 à 80\,ms, imperceptible.

\subsection{Limites identifiées}

\paragraph{1. FIRE partiellement entraîné.}
Les données de collecte pour le geste FIRE ont été capturées principalement
avec la main gauche. En landmarks normalisés, la main droite effectuant le
même geste produit un vecteur approximativement symétrique
($x_i \to -x_i$), qui constitue une forme différente pour le réseau.
La classification FIRE est donc moins fiable avec la main droite.

\paragraph{2. Actions simultanées impossibles.}
La classification étant unimodale (une classe par frame), le système ne peut
envoyer qu'une commande à la fois. Tirer (\texttt{FIRE}) pendant un déplacement
(\texttt{LEFT} ou \texttt{RIGHT}) est impossible avec l'architecture actuelle.

\paragraph{3. NEUTRAL difficile à distinguer.}
La classe NEUTRAL (main au repos) partage des configurations de landmarks
proches avec FIRE (doigts semi-fléchis $\approx$ poing) et LEFT (position
intermédiaire). Cela génère 2 des 2 erreurs de validation et quelques faux
positifs en jeu réel.

%% ============================================================
\section{Perspectives d'Amélioration}
%% ============================================================

\subsection{Court terme — Classes composites pour tirer en se déplaçant}

La limitation jouabilité la plus impactante est l'impossibilité de tirer
en se déplaçant. La solution la plus simple est d'ajouter 2 nouvelles classes
d'entraînement :

\begin{center}
\begin{tabular}{lll}
\toprule
\textbf{Classe} & \textbf{Geste} & \textbf{Commandes envoyées} \\
\midrule
\texttt{LEFT\_FIRE}  & Poing incliné à gauche & LEFT + FIRE simultanés \\
\texttt{RIGHT\_FIRE} & Poing incliné à droite & RIGHT + FIRE simultanés \\
\bottomrule
\end{tabular}
\end{center}

La transition \texttt{LEFT} $\to$ \texttt{LEFT\_FIRE} consiste uniquement à
fermer le poing sans déplacer le bras — geste naturel, peu fatigant, et
géométriquement très différent de tous les autres en landmarks.

\subsection{Court terme — Robustesse FIRE deux mains}

Collecter le geste FIRE avec les deux mains à parts égales lors de la prochaine
session de collecte. Alternativement, générer les échantillons main droite par
augmentation de données ($x_i \to -x_i$, symétrie axiale), sans recollecte.

\subsection{Moyen terme — LSTM pour gestes dynamiques}

Entraîner sur des séquences temporelles de landmarks (5 à 10 frames)
plutôt que sur une frame isolée. Cela permettrait de reconnaître des
\emph{gestes dynamiques} (balayage de gauche à droite, fermeture rapide du poing)
distincts des gestes statiques, réduisant les ambiguïtés géométriques.

\begin{lstlisting}[caption={Architecture LSTM envisag?e pour gestes dynamiques.}]
model = keras.Sequential([
    keras.Input(shape=(10, 63)),     # 10 frames x 63 coords
    keras.layers.LSTM(128, return_sequences=True),
    keras.layers.LSTM(64),
    keras.layers.Dense(32, activation='relu'),
    keras.layers.Dense(n_classes, activation='softmax'),
])
\end{lstlisting}

\subsection{Long terme — Deux mains simultanées}

MediaPipe supporte jusqu'à 2 mains simultanément. En traitant les deux,
on obtient naturellement 2 canaux de contrôle indépendants : par exemple,
main gauche = direction continue, main droite = tir. Cela résout le problème
de simultanéité de manière naturelle et intuitive.

%% ============================================================
\section{Conclusion}
%% ============================================================

Ce projet a permis de construire et déployer un pipeline complet de vision par
ordinateur appliqué au jeu vidéo. Les conclusions techniques principales sont :

\begin{enumerate}
  \item \textbf{La représentation prime sur la complexité du modèle.}
        Passer de $12\,288$ pixels à $63$ landmarks normalisés a fait passer
        la précision de $34.3\%$ à $99.6\%$ avec un modèle 66 fois plus petit.

  \item \textbf{La généralisation dépend des données, pas du dataset.}
        Un modèle entraîné sur 8\,000 images à fond blanc (Kaggle) ne fonctionne
        pas en conditions réelles. Nos 2\,100 landmarks, collectés dans notre
        propre environnement, donnent un modèle robuste et déployable.

  \item \textbf{L'ingénierie système est aussi importante que le modèle.}
        Le threading Python, le hold timer Node.js et les cooldowns sont des
        décisions d'ingénierie — sans elles, un bon modèle donnerait une
        mauvaise expérience utilisateur.

  \item \textbf{Les limites sont comprises et documentées.}
        FIRE moins robuste avec la main droite, impossibilité de tirer en se
        déplaçant — ces limitations sont connues, analysées et des solutions
        concrètes sont proposées.
\end{enumerate}
